# ビリヤード得点アプリ「5-9(ジャパン)」 開発引き継ぎ書

> このドキュメントは Claude との設計会話で固まった仕様をまとめたものです。
> 実装時はこのドキュメントを正として進めてください。不明点があれば実装前に確認してください。

## 0. 前提

- **対象デバイス**: タブレット(台の横に固定設置)、横向き想定
- **利用シーン**: 友人とのカジュアル対戦(3人〜)
- **技術スタック**: Next.js + Vercel + Supabase
- **配信**: LINEを配布導線として使う(LINE自体の置き換えは目指さない)
- 今回スコープは「スコア入力画面」のみ。点数推移画面(グラフ表示)は別タスク。

---

## 1. ゲームルール(5-9 / ジャパン)

基本ルールは9ボールと同じ(若い番号から順に当てて崩す)。3人以上でプレイ可能。

### 1.1 点球・最終球の設定

- **点球(てんきゅう)**: 特殊球として扱う番号。デフォルトは5番だが、**プレイヤーのレベルに応じて卓ごとに変更可能**にする(UI上で番号を選べるようにする)
- **最終球**: 9番または10番を選択可能(9ボール式 / 10ボール式の違いのみ。それ以外のルールは完全に同じ)
- **重要**: MVPでは「今の点球が何番か」をプレイ画面に常時表示する必要はない(後述の通り、得点入力はボール番号ではなく点数ボタンで行うため、プレイヤーは点球の番号を覚えていれば良い)

### 1.2 得点ルール

| 状況 | 獲得点数(対戦相手1人につき) |
|---|---|
| 通常球・コーナーポケット | 1点 |
| 通常球・サイドポケット | 2点(2倍) |
| 点球 or 最終球・コーナーポケット | 2点 |
| 点球 or 最終球・サイドポケット | 4点(2倍) |

- 得点は **「1点」「2点」「4点」の3種類のみ**。MVPでは「何番のボールを入れたか」は記録せず、点数ボタンを直接押す方式とする。
  - **将来拡張の余地を残すこと**: 個別球の記録(どの番号を入れたか)を後から追加できるよう、データ構造はボール番号を保持できる形にしておくと良い(例: `shot` テーブルに `ball_number` カラムを将来追加できるようにする)。MVPでは `point_value`(1/2/4)のみ記録すれば良い。

- **ゼロサム方式**: ボールを入れたプレイヤーは、その時点で卓についている**対戦相手全員からそれぞれ**該当点数を受け取る。
  - 例: 3人プレイ中に通常球をコーナーに入れた → 他の2人からそれぞれ1点ずつ、合計2点獲得(相手は1人1点ずつ減る)
  - 人数が変わればもらえる合計点も自動的に変わる(後述「プレイヤー追加」参照)
  - **全プレイヤーの得点合計は常に0になる**(検算に使える)

```
// 得点付与の擬似コード
function addPoint(currentPlayerIdx, pointValue, activePlayers):
  for each player in activePlayers:
    if player == currentPlayerIdx:
      player.score += pointValue * (activePlayers.length - 1)
    else:
      player.score -= pointValue
```

### 1.3 マスワリ(ボーナス)

- ノーミスでラックを上がりきると「マスワリ」と呼ばれるボーナスが発生し、**そのラックで稼いだ純増点が2倍**になる。
- **判定ロジック**: そのラックで得点した(1球以上入れた)プレイヤーが **1人だけ** であれば、マスワリの可能性がある → ラック終了時に「マスワリでしたか?」と確認する。
  - **2人以上が得点していたら、その時点でマスワリは確定で不成立**。確認すら出さず、そのまま通常終了とする。
  - 「次のプレイヤー」ボタン(後述)を押した回数や、個人がミスしたかどうかではなく、**そのラックの得点者の人数**で判定すること(これが正しいロジックに至るまで何度か試行錯誤したので注意)。

```
function isMasuwariPossible(rackShotsByPlayer):
  scorers = count of players where rackShotsByPlayer[player].length > 0
  return scorers <= 1
```

- マスワリ確定時は、そのラックの各プレイヤーの純増減点(`rackDelta`)を2倍にしてから合計点に反映する。

### 1.4 ラック・区切りの構造

- 1ラック = ブレイクから最終球を上がるまでの1ゲーム
- **10ラックで1区切り**だが、10ラック終わっても明示的な終了操作なしに**そのまま次の10ラックへ継続**する(ハードな区切りで止めない)
- 個人成績は**セッションがいつ終わっても残るよう、継続的に保存**すること(「ゲーム終了」操作有無に依存しない設計にする)

## 2. プレイヤー管理

- 開始時は3人(人数は可変)
- **いつでもプレイヤーを追加可能**(例: 5ラックまで3人、6ラック目から4人)
- **削除機能は不要**(過去のスコア履歴・ゼロサム計算が壊れるため)
- 追加操作は **誤タップ防止のため確認ダイアログを挟む**(「メンバーを追加しますか?」)
- **ラック進行中(得点ボタンか「次のプレイヤー」を1回でも押した後)は追加不可**にし、ラックが終了する(「次のラック」を押す)までグレーアウトする

---

## 3. 打順(手番)管理

**重要な設計判断**: 「打順の並び(`turnOrder`)」と「現在の手番(`currentPlayer`)」を別の状態として持つこと。これにより、打順を入れ替えても今の手番がいきなり変わらないようにできる。

```
state:
  turnOrder: number[]       // 例 [0, 1, 2] = プレイヤー0→1→2→0... の順で回る
  currentPlayerIdx: number  // 実際に今打っている人

function passTurn():        // 「次のプレイヤー」ボタン
  pos = turnOrder.indexOf(currentPlayerIdx)
  currentPlayerIdx = turnOrder[(pos + 1) % turnOrder.length]

function reorder(newTurnOrder):  // 打順変更モーダルでの並び替え
  turnOrder = newTurnOrder
  // currentPlayerIdx には一切触れない
```

### 3.1 通常の進行

- 自分の番でミス(または単に得点なし)をしたら「次のプレイヤー」ボタンで手番を渡す
- 入れている間は自分の番が続く(連続して得点ボタンを押せる)

### 3.2 打順変更モーダル(いつでも呼び出し可能)

- **今の手番のプレイヤーは常に1番目に固定**され、動かせない(リスト上「(手番中)」と表示し、上下移動ボタンは出さない)
- それ以外のプレイヤーは手動で上下入れ替え可能
- **「ランダムに並び替え」ボタン**: 今の手番の人を除いた他のプレイヤーをランダムシャッフルする
  - **直前と全く同じ並びになった場合は再抽選する**(他のプレイヤーが2人以上いる場合のみ。1人しかいない場合はシャッフルしようがないのでそのまま)
- 「決定」を押すまで実際の状態には反映しない(モーダル内でのプレビュー編集 → 確定の2段階)

## 4. スコア入力画面 UI仕様

### 4.1 レイアウト構成(上から順に)

1. **トップバー**: ラック番号 + 「○人目」バッジ(後述) ・・・ 「順番変更」「メンバー追加」「ゲーム終了」ボタン
2. **プレイヤーカード**(横並び、人数に応じて折り返し)
3. **「○○さんの得点を追加」ラベル**
4. **共通得点ボタン(+1 / +2 / +4)** ※プレイヤーごとではなく1セットのみ
5. **「次のプレイヤー」「次のラック」ボタン** ・・・ 右端に「履歴」「取り消し」
6. (条件付き)マスワリ確認パネル / 履歴パネル

### 4.2 得点ボタンは共通化する

各プレイヤーのカードに個別のボタンを置かない。**画面下部に1セットだけ**+1/+2/+4ボタンを置き、押すと**自動的に「現在の手番のプレイヤー」に加点**される。

- ボタンの配色は現在の手番のプレイヤーの色に連動して変える(誰に入るか視覚的にわかるように)
- ボタン上に「○○さんの得点を追加」とテキストでも明示する

### 4.3 ラック内の「○人目」表示

- ラック番号の右に小さいバッジで表示。**初期値は1**(0ではない)
- 「次のプレイヤー」を押すたびに+1
- 「次のラック」でリセットされ1に戻る
- 用途: そのラックで何回手番交代があったかを一目で確認するため

### 4.4 プレイヤーカードの表示内容

- プレイヤー名 + 識別用の小さい色付きドット(常時表示)
- 持ち点合計(大きい数字)
- 「このラック ±X」(そのラックでの純増減点)
- そのラックで獲得した球を**丸数字(①②④)で表示**(古いスコア表記の慣習を踏襲。1点→①、2点→②、4点→④)。ラックが終わるとリセット
- 個別の「点数ボタン」は置かない(4.2参照)

### 4.5 手番の強調表現(重要なデザイン判断)

何度か試行錯誤の末、以下の方式に決定:

- **通常時、全プレイヤーのカードはニュートラル(グレー系)背景**で統一する
- **今の手番のプレイヤーのカードだけ**、そのプレイヤー固有の色で背景・枠線・文字色が変わり、「手番」バッジが表示される(軽く拡大してもよい)
- これにより「全体は落ち着いた配色」かつ「誰の番か一瞬でわかる」を両立する
- 名前横の色ドットだけは非手番時も常に表示し、プレイヤーの色を見失わないようにする

### 4.6 カードの並び順

- 打順(`turnOrder`)に応じて、カードの横並び順も連動して変わる(左から右へ実際のプレイ順になる)
- 打順変更モーダルで並びを変えたら、カードもリアルタイムで並び替わる

### 4.7 「次のラック」ボタン

- **誰も得点していない状態(そのラックでの得点者0人)では押せない**(グレーアウト)
- 押すと:
  - 得点者が2人以上 → マスワリは不成立確定なので、確認なしでそのまま次のラックへ
  - 得点者が1人のみ → 「このラック、マスワリでしたか?」の確認パネルを表示。「はい」でそのラックの純増減点を2倍にしてから次のラックへ

### 4.8 「ゲーム終了」ボタン

- 誤タップ防止のため、**赤系の目立つ色**で表示しつつ、押下後に確認パネル(「本当に終了しますか?この操作は元に戻せません」)を挟む
- 確定すると、最終順位と点数をまとめたサマリー画面を表示する(プレイ画面は非表示にする)

### 4.9 履歴パネル

- 「履歴」ボタンを押した時だけ表示(デフォルト非表示)
- 表示内容(新しい順):
  - 得点: 「プレイヤー名 ①/②/④」
  - 手番交代: 「プレイヤーA → プレイヤーB」(※「ミス」という言葉は使わない。得点が入らなかったことが必ずしもミスとは限らないため)
  - ラック終了: 「ラックN 終了(・マスワリ!)」+ 各プレイヤーの最終増減点
  - 打順変更: 「打順を変更」

### 4.10 取り消し(undo)

- 直前の1操作を取り消せる機能。**頻繁には使わないため、目立たない小さいアイコンボタン**として、「次のプレイヤー」「次のラック」ボタン行の**右端**に配置する
- 得点追加・手番交代・ラック終了・打順変更・打順変更モーダルを開いた操作、それぞれを取り消し対象にする(状態のスナップショットを都度保存しておく方式で実装可能)

## 5. 廃止・スコープ外にしたルール

- **「ハーフ」ルール**(最終球を入れた後、手玉を所定の位置に戻さないと最終球の点数が半分になる)は**現在ほぼ使われていないため実装不要**
- プレイヤーの**削除機能は実装しない**(追加のみ)
- 個別球(何番を入れたか)の記録は将来拡張用にデータ構造だけ意識し、MVPでの記録・表示は不要

---

## 6. データ永続化について

- セッションが「ゲーム終了」されたかどうかに関わらず、**プレイヤーごとの成績は継続的に保存**する(個人成績を後から見返せるようにするため)
- Supabaseのテーブル設計イメージ(叩き台):
  - `sessions`: セッション(卓)の単位
  - `session_players`: セッションに参加するプレイヤー(参加タイミングも記録できると良い)
  - `racks`: ラック単位の記録(ラック番号、マスワリ有無、各プレイヤーの純増減点)
  - `shots`: 得点イベントのログ(将来的にボール番号を追加できる形に)
  - `turn_order_events` または `action_log`: 履歴パネル表示用のイベントログ

## 7. 点数推移画面(2画面目)

### 7.1 画面遷移

- スコア入力画面と推移画面は**タブで切り替える**(画面上部に「スコア」「推移」のセグメントコントロール)

### 7.2 表示の主役

- **折れ線グラフが主役**。ラックごとの点数表は補助的な位置づけとし、小さめのフォント・控えめなスタイルで下に添える。

### 7.3 グラフ仕様

- X軸: ラック番号 / Y軸: 各プレイヤーの累計持ち点
- 1プレイヤー = 1本の折れ線。**色だけでなく線種(実線/破線/点線など)でも区別**すること(色の判別がしづらい環境でも追えるように)
- 凡例はグラフ外に独自実装(色+線種のサンプル表示)
- **0点のライン(収支トントンのライン)を強調表示**する。デフォルトのグリッド線より太く・濃い色で描画し、「0」のラベルを左端に添える。ゼロサムゲームなので「今、自分がプラスかマイナスか」が一目でわかることが重要。

### 7.4 ページング(最大50ラック対応)

- 横スクロールではなく、**10ラック単位でのページ送り**方式とする(「前の10ラック」「次の10ラック」ボタン)
- デフォルトは**最新の10ラックを表示**(今の調子を確認したいというニーズに合わせる)
- 現在の表示範囲を「ラック X-Y / 全Z」の形式で表示する
- 最大50ラック程度まで遡れることを想定

### 7.5 補助テーブル

- グラフの下に、ラックごとの増減点を一覧できる表を配置する(グラフと同じページング範囲に連動して切り替わる)
- 列: ラック番号 + 各プレイヤーの増減点(プラスは色付き、マイナス/ゼロはニュートラルな文字色)
- グラフより視覚的に控えめな小さめフォントサイズにする

### 7.6 配色との一貫性

- スコア入力画面と同じ落ち着いたパレット(テラコッタ・セージ・オーカーなど)を使う

## 8. Claude Codeでの進め方(参考)

- 本ドキュメントをプロジェクトルート(例: `docs/spec.md`)に置き、Claude Codeのセッション開始時に読み込ませると良い
- 実装順序の目安:
  1. Supabaseのテーブル設計・マイグレーション(5章参照)
  2. ゼロサム計算・マスワリ判定・打順管理などのコアロジックを純粋関数として実装(テストしやすくするため。1〜3章の擬似コードがベース)
  3. スコア入力画面のUI実装
  4. 点数推移画面のUI実装(グラフライブラリは Chart.js または Recharts などお好みで)
  5. タブでの画面遷移を実装してつなぐ
- コアロジック(ゼロサム計算・マスワリ判定・打順管理)は仕様が複雑なので、先にユニットテストを書いてからロジックを実装する進め方が事故を防ぎやすい
